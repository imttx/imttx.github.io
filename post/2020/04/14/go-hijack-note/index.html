Go Hijack黑科技

2020年4月14日
| Go
| 阅读

最近在看Go标准库里面的rpc源码，发现了下面一段代码：

 1// ServeHTTP implements an http.Handler that answers RPC requests.
 2func (server *Server) ServeHTTP(w http.ResponseWriter, req *http.Request) {
 3	if req.Method != "CONNECT" {
 4		w.Header().Set("Content-Type", "text/plain; charset=utf-8")
 5		w.WriteHeader(http.StatusMethodNotAllowed)
 6		io.WriteString(w, "405 must CONNECT\n")
 7		return
 8	}
 9	conn, _, err := w.(http.Hijacker).Hijack()	//注意看这里
10	if err != nil {
11		log.Print("rpc hijacking ", req.RemoteAddr, ": ", err.Error())
12		return
13	}
14	io.WriteString(conn, "HTTP/1.0 "+connected+"\n\n")
15	server.ServeConn(conn)
16}

这是一段接管 HTTP 连接的代码，所谓的接管 HTTP 连接是指这里接管了 HTTP 的 TCP 连接，也就是说 Golang 的内置 HTTP 库和 HTTPServer 库将不会管理这个 TCP 连接的生命周期，这个生命周期已经划给 Hijacker 了。

Hijack 介绍

首先看下官方的http.Hijacker接口介绍：

 1type Hijacker interface {
 2	// Hijack lets the caller take over the connection.
 3	// After a call to Hijack the HTTP server library
 4	// will not do anything else with the connection.
 5	//
 6	// It becomes the caller's responsibility to manage
 7	// and close the connection.
 8	//
 9	// The returned net.Conn may have read or write deadlines
10	// already set, depending on the configuration of the
11	// Server. It is the caller's responsibility to set
12	// or clear those deadlines as needed.
13	//
14	// The returned bufio.Reader may contain unprocessed buffered
15	// data from the client.
16	//
17	// After a call to Hijack, the original Request.Body must not
18	// be used. The original Request's Context remains valid and
19	// is not canceled until the Request's ServeHTTP method
20	// returns.
21	Hijack() (net.Conn, *bufio.ReadWriter, error)
22}

Hijack()可以将HTTP对应的TCP连接取出，连接在Hijack()之后，HTTP的相关操作就会受到影响，调用方需要负责去关闭连接。

实现原理

当我们在调用Hijack时，Http Server会将当前连接设置为StateHijacked，并从管理的connection列表中删除，交给使用者自己来管理：

 1// 当前连接设置为StateHijacked
 2func (c *conn) setState(nc net.Conn, state ConnState) {
 3	srv := c.server
 4	switch state {
 5	case StateNew:
 6		srv.trackConn(c, true)
 7	case StateHijacked, StateClosed:
 8		srv.trackConn(c, false) //这里调用trackConn方法
 9	}
10	if state > 0xff || state < 0 {
11		panic("internal error")
12	}
13	packedState := uint64(time.Now().Unix()<<8) | uint64(state)
14	atomic.StoreUint64(&c.curState.atomic, packedState)
15	if hook := srv.ConnState; hook != nil {
16		hook(nc, state)
17	}
18}
19
20//从管理的连接中删除
21func (s *Server) trackConn(c *conn, add bool) {
22	s.mu.Lock()
23	defer s.mu.Unlock()
24	if s.activeConn == nil {
25		s.activeConn = make(map[*conn]struct{})
26	}
27	if add {
28		s.activeConn[c] = struct{}{}
29	} else {
30		delete(s.activeConn, c)
31	}
32}

和正常的HTTP请求区别

为了和正常的HTTP请求对比，写了下面代码：

 1//hijack请求
 2http.HandleFunc("/hijack", func(w http.ResponseWriter, r *http.Request) {
 3		hj, _ := w.(http.Hijacker)
 4		conn, buf, _ := hj.Hijack()
 5		defer conn.Close()
 6
 7		buf.WriteString("hello hijack\n")
 8		buf.Flush()
 9	})
10
11//正常的http请求
12http.HandleFunc("/http", func(writer http.ResponseWriter, request *http.Request) {
13	fmt.Fprintln(writer, "hello http\n")
14})

我们分别来curl请求下这两个接口：

 1$ curl "http://127.0.0.1:9001/hijack" -i
 2hello hijack
 3
 4$ curl "http://127.0.0.1:9001/http" -i
 5HTTP/1.1 200 OK
 6Date: Mon, 17 Feb 2020 03:41:52 GMT
 7Content-Length: 12
 8Content-Type: text/plain; charset=utf-8
 9
10hello http

看出来了吗？乍一看，首先我们能看到使用Hijack的请求返回没有响应头信息。这里我们要明白的是，Hijack之后虽然能正常输出数据，但完全没有遵守http协议。

为什么呢？翻下源码：

 1// Serve a new connection.
 2func (c *conn) serve(ctx context.Context) {
 3	//省略...
 4	serverHandler{c.server}.ServeHTTP(w, w.req)
 5	w.cancelCtx()
 6	if c.hijacked() { //被hijack后直接返回
 7			return
 8	}
 9	w.finishRequest()
10	//省略...	
11}

这是net/http包中的方法，也是http路由的核心方法。调用ServeHTTP方法，如果被Hijack了就直接return了，而一般的http请求会经过后边的finishRequest方法，加入headers等并关闭连接。

使用场景

当不想使用内置服务器的HTTP协议实现时，请使用Hijack。

一般在在创建连接阶段使用HTTP连接，后续自己完全处理connection。符合这样的使用场景的并不多，基于HTTP协议的rpc算一个，从HTTP升级到WebSocket也算一个。

RPC使用

go中自带的rpc可以直接复用http server处理请求的那一套流程去创建连接，连接创建完毕后再使用Hijack方法拿到连接。

 1// ServeHTTP implements an http.Handler that answers RPC requests.
 2func (server *server) servehttp(w http.responsewriter, req *http.request) {
 3    if req.method != "connect" {
 4        w.header().set("content-type", "text/plain; charset=utf-8")
 5        w.writeheader(http.statusmethodnotallowed)
 6        io.writestring(w, "405 must connect\n")
 7        return
 8    }
 9    conn, _, err := w.(http.hijacker).hijack()
10    if err != nil {
11        log.print("rpc hijacking ", req.remoteaddr, ": ", err.error())
12        return
13    }
14    io.writestring(conn, "http/1.0 "+connected+"\n\n")
15    server.serveconn(conn)
16}

客户端通过向服务端发送method为connect的请求创建连接，创建成功后即可开始rpc调用。

WebSocket使用

 1// ServeHTTP implements the http.Handler interface for a WebSocket
 2func (s Server) ServeHTTP(w http.ResponseWriter, req *http.Request) {
 3    s.serveWebSocket(w, req)
 4}
 5
 6func (s Server) serveWebSocket(w http.ResponseWriter, req *http.Request) {
 7    rwc, buf, err := w.(http.Hijacker).Hijack()
 8    if err != nil {
 9        panic("Hijack failed: " + err.Error())
10    }
11    // The server should abort the WebSocket connection if it finds
12    // the client did not send a handshake that matches with protocol
13    // specification.
14    defer rwc.Close()
15    conn, err := newServerConn(rwc, buf, req, &s.Config, s.Handshake)
16    if err != nil {
17        return
18    }
19    if conn == nil {
20        panic("unexpected nil conn")
21    }
22    s.Handler(conn)
23}

websocket在创建连接的阶段与http使用相同的协议，而在后边的数据传输的过程中使用了他自己的协议，符合了Hijack的用途。通过serveWebSocket方法将HTTP协议升级到Websocket协议。

小结

使用Hijack时的一些注意点：

  • Hijack之后的conn需要手动关闭；
  • Hijack之后不能再对w http.responsewriter里面的w写入数据；
  • http.response实现了http.Hijacker接口；
